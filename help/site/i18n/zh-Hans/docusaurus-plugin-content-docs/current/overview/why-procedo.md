---

Procedo 提供了一种结构化的方式来定义和运行操作型工作流，而不需要把每一条执行路径都硬编码进应用程序。

当你需要下面这些能力时，Procedo 会很适合：

- 使用 YAML 编写、容易审查的工作流
- 明确的步骤依赖和可控执行顺序
- 在 .NET 宿主中复用运行时能力
- 面向运维流程的持久化与恢复支持
- 通过插件和自定义步骤进行扩展

## 适合的场景

- 本地自动化
- 部署和发布推进流程
- 人工审批或人工触发流程
- 数据预处理或打包流水线
- 嵌入应用内部的工作流执行

## 它比脚本更有优势的地方

与不断膨胀的 shell 脚本集合或自研编排代码相比，Procedo 在这些方面更清晰：

- 依赖关系
- 运行时状态
- 验证能力
- 可观测性
- 中断后或等待中的流程恢复

## 为什么这里选择 YAML

如果你希望工作流定义具备以下特性，YAML 会非常合适：

- 在 Pull Request 中容易审查
- 与宿主应用的编译逻辑解耦
- 不修改运行时代码也能调整流程
- 团队中不止一位工程师都能看懂

## 取舍

Procedo 非常适合“定义明确的工作流执行”，但它并不打算解决所有编排问题。

当前比较实际的边界包括：

- 单节点运行时模型
- 基于文件的持久化
- 相比通用图合并系统，模板组合能力更克制

对于本地自动化、嵌入式编排以及强调可运维性的工作流，这些取舍通常是可以接受的。
